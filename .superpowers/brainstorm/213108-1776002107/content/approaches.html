Architectural Approaches

Choosing how to build the core launcher engine.

1

"Shell & Event Bindings" (Recommended)

A minimal Tauri shell that loads the remote htmlPath. A Rust background service handles auth, sync, and game launching. We inject a small JavaScript bridge that maps #playButton and other IDs to Rust invokes.

  • Pros: Fast, lightweight, matches the requested "ID-based" interaction perfectly.
  • Cons: Requires careful injection of scripts into remote HTML.
2

"Proxy & API First"

The launcher acts as a local server/proxy. The remote HTML is treated as a web app that calls a structured API. Instead of IDs, the HTML must explicitly call invoke('launch_game').

  • Pros: More robust, easier to test and debug, clearer boundaries.
  • Cons: Requires session creators to know the API; doesn't use the "auto-bind" ID magic as much.

Recommendation: Approach 1

  • Most flexible for remote session creators.
  • Minimal "magic" — just HTML with standard IDs.
  • Fits the "Ultra Optimized" goal by doing less.
